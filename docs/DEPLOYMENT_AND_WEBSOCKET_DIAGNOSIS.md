## Diagnosis answers (Aâ€“H)

### A. Why the pnpm frozen-lockfile error happens

Vercel runs `pnpm install` with **`--frozen-lockfile`** by default (via `installCommand` in `apps/web/vercel.json`). pnpm compares every workspace `package.json` to `pnpm-lock.yaml` importers. When specifiers differ, install aborts with `ERR_PNPM_OUTDATED_LOCKFILE`.

Commit **`2cb7c62`** (`fix(web): repair Vercel build for monorepo workspace packages`) modified `apps/web/package.json` but **did not include `pnpm-lock.yaml`** in the commit (4 files changed; lockfile absent).

### B. Which package manifest change is absent from `pnpm-lock.yaml`

On **`main` at `2cb7c62`**, committed lockfile vs committed `apps/web/package.json`:

| Package | `apps/web/package.json` (committed) | `pnpm-lock.yaml` importers.apps/web (committed) |
|---------|--------------------------------------|--------------------------------------------------|
| `eslint` | `^8.57.0` | `^9.9.0` â†’ resolves to `9.34.0` |
| `eslint-config-next` | `14.2.35` | `14.2.5` â†’ resolves with eslint 9.x |

These devDependency specifier changes are sufficient to trigger the Vercel error. Script-only changes (`build:deps`, `build`) do not affect the lockfile.

*(Local working tree may have an uncommitted lockfile fix for eslint; that fix is **not on `origin/main`** and therefore does not help production builds.)*

### C. Which pnpm version should be standardized on and why

| Source | Declared / detected version |
|--------|----------------------------|
| Root `package.json` `packageManager` | `pnpm@9.7.0` |
| Vercel build log | Lockfile v9 generated by **pnpm 10.x** |
| GitHub Actions `.github/workflows/ci.yml` | `pnpm/action-setup@v4` with **`version: 9`** |
| `README.md` prerequisites | "pnpm 9+" |

**Recommendation for Phase 2:** Standardize on **one pinned pnpm 10.x** release (e.g. latest stable 10.x supported by Corepack), because:

1. Vercel already infers the committed lockfile was produced with pnpm 10.x.
2. Lockfile format version `9.0` is used by both pnpm 9 and 10, so regeneration with pnpm 10 is compatible.
3. Mismatch between `packageManager` (9.7.0) and actual generator (10.x) causes Corepack/Vercel warnings and non-reproducible installs.

Also update **CI** (`ci.yml`) to the same pinned 10.x version. Node **20** is consistent across CI and README ("Node.js 20+"); no `.nvmrc` pins a different version.

### D. Is `apps/server` deployed by the current Vercel configuration?

**No â€” not by anything in this repository's Vercel setup.**

Proof chain:

1. Vercel Root Directory = `apps/web`; build filter = `@kouppi/web` only.
2. No `vercel.json`, serverless function, or WebSocket route under `apps/server` or `apps/web/app/api`.
3. `apps/server` requires a long-lived Node process (`httpServer.listen`, `setInterval` matchmaking every 2s).
4. `README.md` Â§ "Game Server (Production)" explicitly states the server **must run on a persistent host (not Vercel serverless)**.
5. `Procfile` defines a `server` process but there is **no** accompanying platform manifest in the repo proving it is deployed anywhere.

**Cannot confirm from repo alone** whether an operator manually deployed `apps/server` to an external host; there is no URL, Dockerfile, or IaC checked in. `VercelLink.txt` (local, untracked) lists only Vercel frontend URLs.

### E. Can the existing Socket.IO server work on Vercel as currently written?

**No**, not without substantial re-architecture.

Reasons from code:

| Requirement | Current implementation | Vercel constraint |
|-------------|------------------------|-------------------|
| Persistent HTTP + WebSocket | `createServer` + `socket.io` on same port | Serverless functions are ephemeral; no always-on listener in this repo |
| In-memory rooms / queue | `rooms.ts`, `career/queue.ts` | Not shared across instances; lost on cold start |
| Periodic matchmaking | `setInterval(..., 2000)` in `server.ts` | No guaranteed background timer in serverless |
| SQLite file DB | `DATABASE_PATH` on mounted volume | Vercel filesystem is not a durable SQLite host for this pattern |
| Packaging | `node dist/server.js` standalone | No Vercel WebSocket Function wrapper exists in codebase |

Optional Redis (`REDIS_URL`) helps **multi-instance** coordination but does not by itself make the current standalone server deployable on Vercel.

### F. Exact public URL / path / protocol the frontend attempts in production

Assuming a typical Vercel deployment at `https://kouppi-web-nektarios-is-projects.vercel.app` (from `README.md` / `VercelLink.txt`) and **no** `NEXT_PUBLIC_SERVER_URL` set at build time:

#### Casual multiplayer (primary failure path)

| Field | Value |
|-------|-------|
| **Resolved URL** | `http://localhost:4000` |
| **Path** | `/socket.io/` (default) |
| **Protocol** | `ws://localhost:4000/socket.io/...` (WebSocket-only transport) |
| **Code path** | `remoteGameStore.connect()` â†’ `io(serverUrl, { transports: ["websocket"] })` |

The browser tries to open WebSocket to the **user's own machine**, not Vercel â€” connection fails with Socket.IO `"websocket error"`.

#### If `NEXT_PUBLIC_SERVER_URL` were set at build time

Client would use that exact origin (must be `https://` in production for secure pages). Path remains default `/socket.io/` unless configured (it is not today).

#### Career / friends / auth (different fallback)

When `NEXT_PUBLIC_SERVER_URL` is unset, runtime fallback is:

```
window.location.origin.replace(':3000', ':4000')
```

On Vercel production URLs there is **no `:3000` substring**, so this resolves to **`https://<vercel-host>`** (same origin as the Next.js site). That host serves HTML/SSR, **not** Socket.IO â€” connection would also fail (often same `"websocket error"` or HTTP upgrade failure).

#### Leaderboard REST

Uses `NEXT_PUBLIC_API_URL || "http://localhost:4000"` â€” separate env name; same localhost fallback problem.

### G. Immediate root cause of the WebSocket error

**Primary root cause:** There is **no reachable Socket.IO server** at the URL the multiplayer client uses in production.

**Contributing factors (verified in code):**

1. **`apps/server` is not deployed** via the documented Vercel pipeline (Â§D).
2. **`NEXT_PUBLIC_SERVER_URL` is not documented in-repo** with a production value; without it, `remoteGameStore` hard-falls back to **`http://localhost:4000`** (Â§F).
3. **Inconsistent URL helpers** â€” career/friends use origin replace; multiplayer uses localhost fallback â€” so different features fail in different ways, but multiplayer lobby is unambiguously wrong on Vercel.
4. **`transports: ["websocket"]` only** in `remoteGameStore` â€” no polling fallback; fails fast when upgrade is impossible (expected when target is not a Socket.IO server).
5. Error surfacing is generic: `Connection failed: ${err.message}` â†’ user sees **"Connection failed: websocket error"** without indicating missing env or undeployed backend.

This is **Case A** (server not deployed / client points to invalid endpoint), not Case C (server deployed but wrong URL in env) â€” unless an undocumented external server exists and env was mis-set; the repo provides no evidence of such a deployment.

### H. Long-term architecture recommendation

| Option | Fit for KOUPPI today | Notes |
|--------|----------------------|-------|
| **A. URL/config correction only** | **Insufficient alone** | Still requires a running `apps/server` somewhere public. |
| **B. Separately hosted persistent Socket.IO server** | **Recommended immediate path** | Matches `README.md`, `Procfile`, and current code (`server.ts`, in-memory state, timers). Deploy `apps/server` to Railway/Render/Fly/Heroku/VPS; set `NEXT_PUBLIC_SERVER_URL` on Vercel; set `CORS_ORIGIN` on server to Vercel origin(s). |
| **C. Vercel Functions + WebSockets + Redis** | **Not ready; large migration** | Would require rewriting startup model, shared state (Redis mandatory), matchmaking/timer strategy, and SQLite hosting â€” out of scope for a minimal fix. |
| **D. Other** | â€” | e.g. managed realtime (Ably/Pusher) would replace Socket.IO â€” architecture rewrite. |

**Minimum production stack (verified):**

```
Browser â†’ Vercel (Next.js static/SSR)     NEXT_PUBLIC_SERVER_URL â†’ https://<game-server-host>
Browser â†’ Persistent Node host (apps/server)   CORS_ORIGIN â†’ https://<vercel-frontend-origin>
Optional: REDIS_URL for multi-instance rooms/presence
```

**Scaling limitation to disclose:** Without `REDIS_URL`, rooms, matchmaking queue, and friend presence are **single-process in-memory**. Multiple server instances without Redis will split brain.
